

当前实现不是把整首歌一次性塞进模型，而是先缓存整曲 log-mel，再按歌曲切成高度重叠的训练窗口；谱面也被编译成与窗口对齐的 token 序列。接下来我会直接读取现有缓存 manifest 和模型实例统计，给出当前有效歌曲数、窗口数、增强后样本数与精确参数量。

**结论**
当前项目采用“整曲音频转 log-mel 缓存，再按滑窗生成音频张量和谱面 token 张量”的流程。

按当前缓存与配置实际统计：

| 项目 | 当前规模 |
|---|---:|
| 原始 `maidata.txt` | 1,566 份 |
| 音频与谱面成功配对进入缓存扫描 | 1,530 首 |
| 最终可用于 Master 训练 | 1,424 首 |
| 训练歌曲 | 1,211 首 |
| 验证歌曲 | 213 首 |
| 训练基础窗口 | 159,510 个 |
| 8 方向旋转后训练样本 | 1,276,080 个 |
| 模型参数量 | 19,062,528，约 19M |

数据量和 19M 模型大致处于“可以训练，但仍需警惕过拟合”的范围。不能因为增强后有 127 万个样本，就认为数据非常充足，因为其中大量窗口高度重叠，8 倍增强也没有增加新的音乐。

## 数据转换流程

### 1. 原始数据

每首歌通常是：

```text
歌曲目录/
├── track.mp3
└── maidata.txt
```

当前默认只读取：

```yaml
难度编号: 5
```

即 `&inote_5` 的 Master 谱面，见 `config.yaml:55-57`。

缓存构建时还会：

- 排除“宴会場”歌曲。
- 跳过没有 Master 谱面的歌曲。
- 排除违反当前严格谱面规则的歌曲。
- 当前缓存里有 73 首因严格规则被拒绝。

相关逻辑在 `src/chart_cache.py:163-189`。

### 2. 音频变成 log-mel

`track.mp3` 首先由 `torchaudio` 读取，然后：

1. 多声道取平均，变为单声道。
2. 重采样到 22,050 Hz。
3. 计算 MelSpectrogram。
4. 对 Mel 能量执行 `log(mel + 1e-6)`。
5. 保存为 `.cache/charts/*.npy`。

当前参数是：

```text
sample_rate = 22050
n_fft       = 1024
hop_length  = 256
n_mels      = 80
```

因此整首歌会先变成：

```text
[80, 整曲时间帧数]
```

相关代码在 `src/mel_cache.py:72-92`，配置在 `config.yaml:13-24`。

这里没有做 Whisper 原版常见的裁剪、动态范围限制或全局归一化，主要就是普通的自然对数 Mel 能量。

### 3. 整曲切成音频滑窗

每个训练样本包含固定的：

```text
[80, 3000]
```

即 80 个 Mel 频带、3000 个时间帧。

实际音频长度约为：

```text
3000 × 256 / 22050 ≈ 34.83 秒
```

窗口中的区域定义是：

```text
相对 0s                        34.83s
     |----------------------------|
           6s      12s      22s
           | 前缀 | 目标10秒 |
```

具体含义：

- `0..6s`：只有音频上下文，没有谱面 token。
- `6..12s`：谱面历史前缀，模型可见但不计算 loss。
- `12..22s`：真正的训练目标，计算 loss。
- `22..34.83s`：只有未来音频上下文。
- 训练窗口每次向后移动 1 秒。

例如目标区从歌曲第 30 秒开始时：

```text
逻辑音频窗口起点 = 30 - 12 = 18 秒
音频窗口         = 18 .. 52.83 秒
谱面前缀         = 歌曲 24 .. 30 秒
训练目标         = 歌曲 30 .. 40 秒
```

歌曲开头和结尾不足的部分直接补零，目标区始终固定在窗口的 `12..22s`，不会移动目标位置。

对应代码：

- 窗口坐标：`src/chart_cache.py:193-220`
- 读取并补零：`src/dataset.py:336-360`
- 参数配置：`config.yaml:26-39`

### 4. 谱面变成 token

`maidata.txt` 经 parser 解析成带时间的谱面 frame，然后每个 frame 通过 `encode_frame()` 编码成 token。

一个窗口的 token 大致是：

```text
SOS
前缀区 frame token...
目标区 frame token...
EOS
PAD...
```

其中 frame token 会表达：

- 时间戳 TS
- frame 开始和结束
- Tap、Hold、Slide、Touch 等音符类型
- 轨道/Touch 区域
- 持续时间
- Slide 形状和参数
- Break、EX 等属性

当前词表大小：

```text
3072
```

窗口最大长度：

```text
2048 token
```

缓存中的 token 使用 `uint16` 保存；读取样本时转换成 PyTorch `int64`。

当前实际统计：

```text
平均窗口长度：约 411 token
中位数：410 token
最大值：1307 token
```

所以当前真实数据离 `2048` 上限还有一定余量。

相关代码：

- 谱面窗口编码：`src/chart_cache.py:199-220`
- token mmap 读取：`src/dataset.py:106-139`
- token 转 `int64` 和 padding：`src/dataset.py:362-379`

### 5. loss mask

模型能看到前缀，但不要求预测前缀。`loss_mask` 只覆盖：

```text
目标区第一个 FRAME_START
...
EOS
```

如果一个目标窗口里没有音符，则只监督 EOS。

这意味着训练实际上是：

```python
decoder_input = tokens[:, :-1]
target = tokens[:, 1:]
```

但交叉熵只在目标区和 EOS 上计算。

当前训练集在不计算旋转重复的情况下，大约有：

```text
40,761,562 个受监督 token
```

8 方向增强后，每轮会呈现约：

```text
326,092,496 个受监督 token
```

但后者主要是同一谱面的旋转版本，不是新增音乐内容。

### 6. DataLoader 最终输出

批大小目前是 24，所以每个 batch 大致为：

```text
mel:         [24, 80, 3000]   float32
tokens:      [24, 2048]       int64
mask:        [24, 2048]       bool
loss_mask:   [24, 2048]       bool
```

音频进入 encoder 后，第二层卷积以 stride 2 下采样，变成：

```text
[24, 1500, 384]
```

decoder 输入最多是：

```text
[24, 2047]
```

然后通过 cross-attention 读取音频特征。

对应代码：

- batch 拼接：`src/dataset.py:392-417`
- encoder：`src/model.py:171-201`
- decoder：`src/model.py:204-246`
- DataLoader：`src/train.py:173-186`

## 数据增强

有数据增强，而且目前固定扩成 8 倍。

### 旋转增强

训练集经过：

```python
RotatedDataset(base_train_ds, rotations=8)
```

它对每个谱面生成顺时针 `0..7` 步的全部圆周旋转：

```text
原 Tap lane 1 -> lane 2 -> ... -> lane 8
Slide 起终点同步旋转
Touch 位置按规则旋转
```

但有两个关键点：

- 只旋转谱面 token。
- 音频完全不变。

这是合理的，因为 maimai 的整体圆周方向与音乐内容通常没有必然关系，可以让模型少记忆“某种节奏只能落在某条轨道”。

验证集不做旋转增强，避免验证指标被重复样本稀释。

对应代码在 `src/dataset.py:146-166`，训练包装在 `src/train.py:132-134`。

### 滑窗也具有增强效果

训练窗口每 1 秒移动一次，而每个窗口有约 34.83 秒音频。因此相邻两个样本高度重叠：

```text
窗口 A：0.00 .. 34.83s
窗口 B：1.00 .. 35.83s
```

同一段音乐会在不同相对位置和不同上下文中反复出现。这有助于模型学习时间平移下的稳定性，但不能把这些窗口视为完全独立的新样本。

### 当前没有的增强

代码中目前没有看到这些常见音频增强：

- 随机增益
- 加噪声
- EQ 或滤波
- SpecAugment 时间遮挡/频率遮挡
- 时间伸缩
- 音高变化
- 混音或背景噪声
- 随机裁剪相位
- 音频和谱面同步的速度变换

也没有随机选择旋转；当前是每个基础窗口在每轮都完整遍历 8 个方向。

## 训练集规模

### 原始歌曲规模

当前磁盘统计：

```text
charts/             约 4.6 GB
.cache/charts/      约 5.8 GB
当前训练索引        约 159 MB
```

文件与缓存统计：

```text
原始 maidata.txt                 1566
音频/谱面有效配对来源            1530
最终可用 Master 严格规则谱面      1424
```

1,566 与 1,530 的差值主要来自缺少 `track.mp3` 等无法形成有效音频/谱面对的目录。

1,530 到 1,424 的差值包括：

- 73 首严格规则违规。
- 宴会場排除。
- 没有对应 Master 难度等情况。

### 按歌曲切分

训练不是随机切窗口，而是先按歌曲划分：

```text
训练：85%
验证：15%
```

在随机种子 42 下，当前得到：

```text
训练歌曲：1211
验证歌曲：213
```

这是正确做法。否则同一首歌的高度重叠窗口可能同时出现在训练集和验证集，造成严重的数据泄漏。

切分逻辑见 `src/train.py:113-134`。

### 窗口规模

当前 1 秒步进索引共有：

```text
全部窗口：187,089
训练基础窗口：159,510
验证歌曲对应的 1 秒窗口：27,579
```

训练集经过 8 方向旋转后：

```text
159,510 × 8 = 1,276,080 samples / epoch
```

按每个基础窗口对应 1 秒的新歌曲位置估算：

```text
全部歌曲总时长约 187,089 秒 ≈ 52 小时
训练歌曲约 44 小时
```

这里的“52 小时”更接近独立音乐内容量。不能把每个窗口的 34.83 秒直接相乘，否则会把重叠部分重复计算约 35 次。

验证实际使用 10 秒步进的独立索引，因此验证窗口量大约在 2,800 个左右，而不是把所有 27,579 个 1 秒步进窗口拿去验证。

## 和模型大小是否匹配

当前模型参数为：

```text
19,062,528，约 19.1M 参数
```

结构是：

```text
Audio encoder:
  state = 384
  heads = 6
  layers = 4

Text decoder:
  state = 384
  heads = 6
  layers = 4

Vocabulary = 3072
Audio context = 1500
Text context = 2048
```

### 我的判断：基本匹配，但数据不算特别宽裕

有利因素：

- 19M 不算大模型。
- 训练歌曲有 1,211 首、约 44 小时。
- 有约 4,076 万个不重复旋转的受监督 token。
- 8 方向旋转能有效降低 lane 记忆。
- 每秒滑窗增加了上下文位置多样性。
- encoder-decoder 架构规模比较克制，只有各 4 层。

风险因素：

- 真正独立的数据单位主要是 1,211 首歌，而不是 127 万个增强样本。
- 相邻窗口有约 97% 的音频重叠。
- 8 个旋转版本共享完全相同的音频和节奏结构。
- 单一 Master 难度的数据分布较窄。
- 音频与谱面可能存在同作者、同系列、同曲风和重复版本，实际多样性低于歌曲数量。
- 从头训练 19M 参数，比使用预训练音频 encoder 更依赖数据质量。
- 500 epoch 上限非常高；按照 127 万增强样本计算，完整跑满会反复暴露同一歌曲很多次。

因此更准确的评价是：

```text
足以训练一个 19M 的实验模型，
足以做整曲过拟合和验证架构是否工作，
也有机会学出可用的节奏/音符模式；
但未必足以从头训练出泛化非常强的自动谱面模型。
```

### 不建议直接增大模型

在目前数据规模下，我不建议先把模型扩到 50M、100M。更大的模型更容易：

- 记住训练歌曲。
- 记住谱师习惯。
- 在 teacher forcing 指标很好，但整曲递推生成变差。
- 增加训练成本，却未必提高跨歌曲泛化。

当前 19M 是比较合理的起点。是否还偏大，应该看以下现象：

| 现象 | 判断 |
|---|---|
| train loss 和 val loss 都高 | 模型、优化或表示能力不足 |
| train loss 持续下降，val loss 很早恶化 | 数据不足或模型过拟合 |
| teacher forcing 很好，整曲生成差 | 暴露偏差/递推误差，不一定是模型太大 |
| 训练集本身都无法生成正确 | 先查训练、EOS、token 和约束，不要缩模型 |
| 少量歌曲能过拟合，全量验证仍差 | 数据多样性或泛化问题 |
| 旋转后的训练指标好，但未旋转验证差 | 增强或 lane 表示存在问题 |

### 最值得关注的问题

当前最需要避免的是把“127 万训练样本”误认为 127 万个独立数据点。更真实的规模描述是：

```text
1211 首独立训练歌曲
约 44 小时独立音频
159,510 个高度重叠的基础窗口
每个窗口 8 个几何旋转版本
约 4076 万个基础受监督 token
19.1M 参数
```

以这个口径看，模型和数据总体是匹配的，但属于中等偏紧，而不是数据极其充足。当前优先级应该是通过按歌曲验证和正式整曲生成 F1 判断是否过拟合，而不是继续扩大模型或继续把滑窗步进调得更密。

这里“token 数”有三种口径：整曲谱面唯一 token、10 秒目标区监督 token、以及包含 6 秒前缀的 decoder 序列 token。为和训练有效数据一致，我先按当前严格规则缓存统计“目标区监督 token”，并分别列出原始不重叠 10 秒窗口、1 秒平滑步进、8 方向旋转后的规模。


以下以**当前严格规则过滤后的 Master 数据**为准，并以真正参与 loss 的“目标区监督 token”作为主要口径。

## 全部有效数据

有效数据共 **1,424 首，约 51.97 小时独立音乐**。

| 阶段 | 样本/窗口数 | 监督 token | 相对原始 | 目标区呈现时长 |
|---|---:|---:|---:|---:|
| 原始，不重叠 10 秒切分 | 19,329 | 4,860,711 | 1× | 53.69 小时 |

| 1 秒平滑步进 | 187,089 | 47,803,456 | 9.83× | 519.69 小时 |

## 如果把前缀 token 也算上

每个 decoder 序列还包括：

- `SOS`
- 相对 `6..12s` 的历史谱面前缀
- 相对 `12..22s` 的目标谱面
- `EOS`

这些前缀 token 可以被模型看到，但不参与 loss。按这个完整 decoder 序列口径：

| 数据范围 | 原始不重叠 | 1 秒步进 | 步进 + 8 旋转 |
|---|---:|---:|---:|
| 全部有效数据 | 7,758,867 | 76,928,678 | 615,429,424 |
| 实际训练集 | 6,613,593 | 65,592,541 | 524,740,328 |

因此“每 epoch 处理多少 token”有两个答案：

```text
实际训练集完整 decoder token：约 5.247 亿
其中真正计算 loss 的监督 token：约 3.261 亿
```

## “多少小时”需要区分

增强不会创造新的音乐：

```text
独立训练音乐始终只有约 44.31 小时
全部有效音乐始终只有约 51.97 小时
```

所谓增强后的几百或几千小时，指的是**每个 epoch 重复呈现给模型的目标区总时长**：

```text
训练集独立音乐：             44.31 小时
1 秒步进后的目标区呈现：    443.08 小时
再乘 8 个旋转：           3544.67 小时/epoch
```

但这 3,544.67 小时不能当作 3,544 小时的独立数据：

- 1 秒步进的相邻目标区有 9 秒重叠。
- 相邻完整音频输入窗口约有 33.83 秒重叠。
- 8 个旋转版本使用完全相同的音频。
- 旋转只改变谱面方向，不改变节奏、音符类型和音乐内容。

最准确的概括是：

```text
原始训练内容：
1211 首 / 44.31 小时 / 414 万监督 token

平滑步进后：
15.95 万窗口 / 4076 万监督 token

再做 8 方向旋转后：
127.61 万样本 / 3.261 亿监督 token/epoch

真正独立的音乐内容：
仍然只有 44.31 小时
```
