# 数据量和模型参数概况

本文只描述当前源码、`config.yaml` 和本机缓存能够确认的状态。数据集会随 `charts/`、MERT 缓存、严格规则和配置变化，旧索引统计不能继续当作当前值。

## 当前结论

当前流程是：整曲音频先经冻结的 `MERT-v1-95M` 提取 75 Hz、768 维时序特征，再按歌曲切成滑窗；可训练模型只包含音频特征归一化、位置编码和 4 层自回归文本 decoder。

| 项目 | 当前值 |
|---|---:|
| 原始 `maidata.txt` | 1,566 份 |
| 原始 `track.mp3` | 1,534 份 |
| 已完成的 MERT `.npy` 缓存 | 599 份 |
| 难度 | `level_idx=5`，Master |
| MERT 窗口 | 2,612 帧，约 34.83 秒 |
| MERT 特征维度 | 768 |
| decoder 最大长度 | 2,048 token |
| 词表大小 | 3,072 |
| 可训练模型参数 | 43,742,208，约 43.7M |
| 冻结特征模型 | MERT-v1-95M，不计入上述参数量 |

当前 `.cache/mert/` 约 8.5 GB，但缓存尚未覆盖全部原始歌曲，且当前没有可复用的 `.cache/chart_index/`。因此不能从现状确认最终可用歌曲数、窗口数、监督 token 数和训练/验证划分规模。此前文档中的 1,424 首、159,510 个基础窗口、1,276,080 个旋转样本等数字属于旧缓存快照，现已删除。

## 数据转换流程

### 1. 冻结 MERT 特征

`src/mert_cache.py` 使用本地 `MERT-v1-95M`：

1. 解码 `track.mp3`，多声道取平均。
2. 使用 Hugging Face processor 转成模型输入。
3. 长音频按 30 秒核心区分块，每侧带 10 秒上下文。
4. 读取冻结 MERT 的 `last_hidden_state`。
5. 拼接为整曲 `(时间帧数, 768)` 特征并缓存为 `.npy`。

当前按 75 Hz 使用这些特征，缓存配置为 `float16`。MERT 只离线提取特征，不参与训练，也不会随 decoder 反向传播。

这与旧实现有本质区别：当前没有 80 维 log-mel 输入、`[80, 3000]` 张量或卷积音频 encoder。

### 2. 滑窗

每个样本直接读取：

```text
[2612, 768]
```

窗口时长为：

```text
2612 / 75 ≈ 34.83 秒
```

区域定义：

```text
相对 0s                        34.83s
     |----------------------------|
           6s      12s      22s
           | 前缀 | 目标10秒 |
```

- `0..6s`：只有音频上下文。
- `6..12s`：真实谱面前缀，decoder 可见但不计算 loss。
- `12..22s`：训练目标，计算 loss。
- `22..34.83s`：未来音频上下文。
- 训练窗口按 1 秒步进。
- 正式推理按 10 秒步进，每窗只提交目标区。
- 歌曲头尾不足的部分补零，并由 cross-attention mask 屏蔽。

例如目标从歌曲第 30 秒开始：

```text
逻辑窗口起点 = 30 - 12 = 18 秒
MERT 窗口    = 18 .. 52.83 秒
谱面前缀     = 24 .. 30 秒
训练目标     = 30 .. 40 秒
```

### 3. 谱面 token

每个窗口的序列为：

```text
SOS
前缀区 frame token...
目标区 frame token...
EOS
PAD...
```

token 表达 frame 时间、Tap、Hold、Slide、Touch、轨道或区域、持续时间、Slide 参数以及 Break、EX 等属性。词表范围当前为 `0..3071`；索引缓存用 `uint16`，读取后转成 PyTorch `int64`。

配置的最大序列长度是 2,048。索引构建时如果真实窗口超过此值，`ChartDataset` 会拒绝截断并报错。

### 4. loss mask

训练采用 teacher forcing：

```python
decoder_input = tokens[:, :-1]
target = tokens[:, 1:]
```

但 loss 只覆盖目标区第一个 `FRAME_START` 到 `EOS` 的连续后缀。没有目标 frame 时只监督 `EOS`。当前还给 `EOS` 使用 5 倍 loss 权重，以缓解结束 token 的类别失衡。

## DataLoader 输出

当前配置批大小为 8，一个完整 batch 的固定形状是：

```text
mel:         [8, 2612, 768]  float32
audio_mask:  [8, 2612]       bool
tokens:      [8, 2048]       int64
mask:        [8, 2048]       bool
loss_mask:   [8, 2048]       bool
```

这里的变量名 `mel` 是历史命名，内容实际是 MERT 特征。模型不会再对时间轴做卷积下采样，cross-attention 直接读取 2,612 个音频特征位置。

## 数据增强与划分

训练集使用 `RotatedDataset`，每个基础窗口固定遍历顺时针 `0..7` 共 8 个整体旋转：

- 只旋转谱面 token，不修改音频。
- Tap lane、Slide 几何和 Touch 区域按规则同步旋转。
- 验证集不旋转。
- 训练/验证先按歌曲划分，避免同一歌曲的重叠窗口泄漏到两侧。

1 秒步进会让相邻完整音频窗口重叠约 33.83 秒，8 个旋转也共享同一音频和节奏。因此“基础窗口数 × 8”不是独立音乐数据量。

当前 `config.yaml` 还设置了：

```yaml
过拟合歌曲数: 5
```

这意味着直接运行 `src/train.py` 时只抽取 5 首完整歌曲，并让训练和验证使用同一批歌曲，用于过拟合排查。它不是正常的全量训练配置。要统计或执行正常的按歌曲 85%/15% 划分，应先把该值设为 `0`。

## 模型结构和参数

当前可训练模型配置：

```text
音频上下文长度 = 2612
文本上下文长度 = 2048
状态维度       = 768
注意力头数     = 12
decoder 层数   = 4
词表大小       = 3072
```

模型没有可训练的 MERT encoder，也没有旧版的 4 层音频 Transformer encoder。音频侧只有：

- 2,612 × 768 的可训练位置编码。
- 一层 768 维 LayerNorm。

文本侧每层包含 causal self-attention、对 MERT 特征的 cross-attention 和 4 倍扩展的 MLP。输出层与 token embedding 权重共享。

参数量由当前代码直接实例化统计：

```text
43,742,208 个可训练参数，约 43.7M
```

其中冻结的 `MERT-v1-95M` 只用于缓存生成，不在训练模型参数、optimizer 状态或 checkpoint 的 `model_state_dict` 中。讨论显存和训练规模时应使用 43.7M；讨论整套系统依赖的预训练能力时，则应同时说明前端使用了约 95M 的冻结 MERT。

## 如何获取新的精确数据量

最终可用歌曲数不是原始文件数，因为缓存构建还会排除：

- 缺少 `track.mp3` 或 MERT 缓存的目录。
- 没有指定 Master 谱面的歌曲。
- “宴会場”歌曲。
- 违反 `constrained_decode.py` 当前严格规则的谱面。

当 MERT 缓存完整并重新构建 chart index 后，以生成目录中的 `manifest.json`、`entries.npy` 和 `tokens.npy` 为准。训练启动日志还会输出实际训练/验证歌曲数、基础窗口数、旋转后样本数、监督 token 数和 EOS 比例。

在这些缓存完成前，不应把任何旧的歌曲数、小时数或 token 总量写成“当前实际统计”。
